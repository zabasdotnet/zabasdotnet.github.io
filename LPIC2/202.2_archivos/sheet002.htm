Montaje de sistemas de archivos Volver
a. Montaje y desmontaje
El comando mount permite montar el sistema de archivos de un periférico de almacenamiento en un directorio local.
Generalmente, este directorio suele dejarse vacío.
Como mínimo hay que pasarle por parámetro al comando mount el periférico que alberga el sistema de archivos y el directorio que constituirá su punto de montaje.
El comando umount realiza la operación inversa.
Acepta como argumento el punto de montaje o el periférico físico que se desmontará.
*Montaje de un sistema de archivos
mount -t tipo_fs -o opciones dispositivo punto_de_montaje
*Opciones y parámetros:
tipo_fs --> Opcional: tipo de sistema de archivos que se desea montar.
opciones --> Opcional: opciones de montaje.
dispositivo --> El periférico que alberga el sistema de archivos que queremos montar, identificado con un archivo especial de bloque.
punto_de_montaje --> El directorio que servirá de punto de montaje para el sistema de archivos montado.
Las opciones más comunes son:
ro (solo lectura)
sync (escrituras síncronas sin pasar por una caché en memoria)
loop (montaje de datos de archivos en lugar del sistema de archivos).
El montaje de un sistema de archivos con la opción sync permite sustituir el uso de la caché por escrituras directas a disco y, de este modo, hacer más fiables las operaciones de escritura.
El comando sync permite vaciar puntualmente la caché de un sistema de archivos que no se haya montado con esta opción de montaje.
*Desmontaje de un sistema de archivos
umount -O opciones dispositivo punto_de_montaje
*Opciones y parámetros:
opciones --> Opcional: opciones de desmontaje.
dispositivo --> Opcional si el punto de montaje se ha definido: el periférico que se desea desmontar. 
punto_de_montaje --> Opcional si el periférico se ha definido: el directorio que sirve de punto de montaje que queremos liberar.
Las opciones más comunes son:
-f --> (force: forzar el desmontaje)
-l --> (lazy: desmontaje perezoso que se hará efectivo cuando todos los recursos utilizados por el montaje hayan sido liberados).
El desmontaje de un sistema de archivos es necesario para realizar su revisión mediante el comando e2fsck.
Por definición, el sistema de archivos montado en / no se puede desmontar debido a que está siempre ocupado.
Se puede forzar la comprobación antes del montaje cuando arranca el sistema desde el comando shutdown.
*Revisión del sistema de archivos raíz antes del montaje
shutdown -F -r now
b. Visualización de los sistemas de archivos montados
El comando mount sin argumentos permite visualizar los sistemas de archivos montados.
Además, cada montaje con éxito genera su línea correspondiente en el archivo /etc/mtab.
La visualización del archivo /proc/mounts devuelve la misma información.
c. Archivo fstab
El archivo /etc/fstab permite definir los sistemas de archivos o los espacios de swap que se montarán automáticamente en el arranque.
De forma opcional también permite definir sistemas de archivos que es posible que se monten, como por ejemplo los periféricos extraíbles.
La sintaxis del comando mount invocado puntualmente se simplificará bastante.
El archivo /etc/fstab debe tener en cada línea el conjunto de elementos necesarios para el montaje de un sistema de archivos.
Estos elementos son el punto de montaje, la definición del espacio de almacenamiento y las opciones de montaje.
Para los espacios de swap la definición del punto de montaje no es aplicable.
El archivo /etc/fstab se compone de una línea por cada sistema de archivos que se montará.
A su vez, cada línea se compone de seis campos obligatorios.
*Formato típico de una línea de declaración de montaje en /etc/fstab:
fs puntodemontaje tipo opciones dump fsck
Los campos se separan por espacios o tabulaciones.
*Archivo /etc/fstab: formato de las líneas de definición de montaje
Número de campo || Campo || Definición
1 || fs || Sistema de archivos, definido por su archivo especial de bloque, su etiqueta o su uuid.
2 || puntodemontaje || Punto de montaje.
3 || tipo || Tipo de sistema de archivos. Obligatoriamente swap para swap, auto o tipo real de sistema de archivos en caso contrario.
4 || opciones || Opciones de montaje. De hecho, son las opciones admitidas por el comando mount.
5 || dump || Opcional. Si el comando dump se utiliza para la copia de seguridad del sistema, este campo tiene que valer 1 para asegurar la copia de seguridad. Si no, su valor por defecto es 0.
6 || fsck || Opcional. En caso de revisión automática del sistema de archivos en el arranque, indica en qué orden se tiene que realizar la comprobación. Valor obligatorio de 1 para el sistema de archivos montado en /, 2 para el resto. 0 para que la revisión no se realice nunca.
*Ejemplo de archivo /etc/fstab en Ubuntu:
Observe que los discos se identifican con su UUID.
# /etc/fstab: static file system information. 
# 
# Use blkid -o value -s UUID to print the universally unique identifier 
# for a device; this may be used with UUID= as a more robust way to name 
# devices that works even if disks are added and removed. See fstab(5). 
# 
# <file system> <mount point>   <type>  <options>       <dump>  <pass> 
proc            /proc           proc    nodev,noexec,nosuid 0       0 
# / was on /dev/sda1 during installation 
UUID=e6ad7ebe-7128-4182-ad58-5416bad59fce /               ext4     
errors=remount-ro 0       1 
# swap was on /dev/sda5 during installation 
UUID=f6c2ca34-eb80-4950-b243-984d9a5a9d92 none            swap     
sw              0       0

Los discos se identifican con su etiqueta.
LABEL=/                 /                       ext3    defaults 
1 1  
LABEL=/boot             /boot                   ext3    defaults 
1 2  
/dev/hda3               swap                    swap    defaults 
0 0 
/dev/cdrom              /mnt/cdrom              udf,iso9660 
noauto,owner,kudzu,ro 0 0 
/dev/fd0                /mnt/floppy             auto 
noauto,owner,kudzu 0 0
El comando mount -a se invoca en el arranque del sistema operativo.
Este comando provoca el montaje de todos los periféricos a los que se hace referencia en el archivo /etc/fstab, a excepción de los que presentan la opción noauto en el cuarto campo.
d. Automontaje
El montaje puede ser una operación difícil de realizar para el operador.
Algunas características opcionales pueden superar cierto grado de conocimiento de las funciones y comandos de montaje.
Los entornos gráficos, GNOME o KDE por ejemplo, administran desde hace ya bastante tiempo el montaje automático de periféricos extraíbles.
La certificación LPI requiere el conocimiento del automontaje, técnica que permite montar un sistema de archivos sobre la marcha en función del acceso que realiza el usuario.
El automontaje está en vías de desaparecer en las estaciones de trabajo autónomas, pero es muy eficaz en los sistemas de archivos distribuidos.
*Configuración del automontaje
El automontaje se basa en dos archivos de configuración (las tablas de automontaje) y en un servicio que comprueba permanentemente si hay que realizar operaciones de montaje.
La primera tabla de automontaje, la tabla maestra, se configura en el archivo /etc/auto.master.
En ella se detalla el conjunto de directorios en los que se aplica el automontaje, es decir, los directorios en los que se podrán realizar los montajes automáticos si una aplicación accede al contenido de este directorio.
Se crearán en este archivo tantas líneas como directorios haya para controlar.
*Formato de la tabla maestra de automontaje (/etc/auto.master)
directorio arch_sec --opción=valor
*Directivas y variables utilizadas:
directorio --> El directorio en el que se controlarán los accesos para ver si hay que proceder al automontaje.
arch_sec --> El archivo de la tabla secundaria que detalla los montajes que se han de realizar en este directorio.
opción --> Opción relacionada con la administración del autofs. Opción común que hay que conocer: timeout. El valor es, por lo tanto, el número de segundos antes de desmontarlo en caso de inactividad.
No hay restricciones en cuanto al nombre de los archivos de la tabla secundaria, aunque por lo general se pone el prefijo «auto.» y se ubican en /etc.
Se necesitarán tantos archivos secundarios como se haya descrito en la tabla maestra.
En la mayoría de los casos, una única tabla secundaria es suficiente.
Cada tabla secundaria corresponde con la carga de un daemon independiente.
*Formato del archivo de tabla secundaria
pmv opciones dispositivo
*Directivas y variables:
pmv --> Punto de montaje virtual: el directorio cuyo acceso realizado por aplicaciones provocará el montaje.
opciones --> Las opciones de montaje, precedidas por un guión y separadas por comas.
dispositivo --> El periférico que se montará.
*Administración del servicio de automontaje
Para que se tenga en cuenta la nueva configuración, hay que reiniciar el servicio.
El script de ejecución del servicio se llama generalmente autofs y se encuentra en /etc/init.d.
Todo cambio en la tabla maestra debe acompañarse de un reinicio del servicio.
*Reinicio del servicio
/etc/init.d autofs restart
*Visualización de la configuración
/etc/init.d autofs status
*Funcionamiento del automontaje
Para comprobar la configuración de su automontaje, siga este procedimiento:
1) Cree un directorio de trabajo. No cree los puntos de montaje, no tienen que existir.
2) Genere los archivos de configuración.
3) Reinicie el servicio.
4) Desde el shell, sitúese en el punto de montaje virtual, el que se ha descrito en el archivo de configuración de la tabla secundaria.
5) Compruebe que el montaje se ha realizado de forma transparente con éxito.
