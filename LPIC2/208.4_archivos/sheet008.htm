Balanceo de carga con un servidor Nginx Volver
El módulo de balanceo de carga (load balancing), http_upstream, permite definir clústeres de servidores a los que redirigir las peticiones.
Se especifica una estrategia de elección del servidor de destino para equilibrar las cargas de los diferentes miembros del grupo.
1. El bloque upstream
El módulo gestiona el tipo de bloque upstream, que define un clúster de servidores:
upstream NombreGrupo { 
}
Cada servidor de un bloque upstream se define con una directiva server:
server Host [parámetros];
*Con:
Host --> se puede especificar el nombre de host, su dirección IP o una ruta de acceso de socket Unix con el prefijo unix:.
Los parámetros permiten especificar las características de utilización del servidor.
*Los principales son:
weight=Nombre
Peso asociado al servidor (1 por defecto).
backup
Este servidor se utilizará solo si no se puede acceder a los servidores primarios.
max_conns=Número
Número máximo de conexiones simultáneas soportadas por el servidor.
max_fails=Número
Número de fallos de conexión al servidor tras los cuales se considerará inaccesible. 0 desactiva el control.
fail_timeout=Duración
Duración en la que un servidor con errores se mantendrá en estado inaccesible, y mediante la cual se aplica el cálculo del número de intentos fallidos, antes del reinicio del ordenador.
La estrategia de elección del servidor en un clúster de servidores es, por defecto, un «round robin» ponderado.
Si todos los servidores tienen el mismo peso (argumento weight), se utilizarán sucesivamente.
Si algunos tienen más peso, se les harán peticiones de forma proporcional a este pero respecto al total del clúster:
*Ejemplo
    upstream servidores { 
        server www1.masoc.com weight=2; 
        server www2.masoc.com weight=1; 
        server www3.masoc.com weight=1; 
        server seguridad.masoc.com  backup; 
    }
*Definición de un clúster de servidores llamado servidores.
Para cada conjunto de 4 peticiones, 2 se asignarán al servidor www1, 1 a www2 y 1 a www3.
El servidor seguridad solo recibirá peticiones si alguno de los tres servidores primarios no está accesible.
La estrategia de elección del servidor se puede modificar mediante diferentes directivas, entre las cuales:
hash Clave;
El balanceo de carga entre los servidores se hace aplicando una tabla de hash a partir de la cadena de caracteres Clave.
ip_hash;
El reparto de la carga entre los servidores se hace respecto de las direcciones IP, a partir de las cuales se efectúa un cálculo de una clave de hash.
Este método garantiza que un mismo cliente se asociará al mismo servidor, excepto si este último queda inaccesible.
2. Utilización de un clúster de servidores
Varias directivas permiten asociar una petición a un clúster de servidores.
También se puede declarar un clúster de servidores como proxy, con la directiva proxy_pass:
proxy_pass Protocolo://NombreClúster;
Las llamadas fastCGI también se pueden enviar a un clúster de servidores.
fastcgi_pass NombreClúster;
