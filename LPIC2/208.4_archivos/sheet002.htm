1. Nginx y los servidores web Volver
Si bien Apache es el servidor web más conocido y el más utilizado en Internet, en los últimos años hemos visto aparecer serios competidores.
A menudo, el primer objetivo de estos nuevos servidores web es el rendimiento.
De hecho, a Apache se le reprocha una cierta lentitud, relacionada con el hecho de que es un servidor antiguo y con sus numerosas funcionalidades, así como un consumo excesivo de recursos cuando se le hacen muchas peticiones.
En 2002, un desarrollador ruso, Igor Sysoev, decidió escribir un programa completamente nuevo, en el ámbito de la gestión de una web rusa muy frecuentada, Rambler.
Tenía que soportar más de 500 millones de peticiones HTTP diarias.
La primera versión de este programa, Nginx (se pronuncia engineX), apareció en 2008.
Escrito en lenguaje C y creado para ser simple y eficaz, este servidor web rápidamente ganó cuota de mercado, sobre todo para sitios con mucho volumen de peticiones (en particular Facebook).
En 2013 representaba alrededor del 15% de los servidores web de Internet, en tercera posición después de Apache (64%) y justo detrás de IIS de Microsoft (17%).
También puede ser servidor proxy de correo (SMTP, POP3 e IMAP4)
Actualmente se utiliza en el campo HTTP.
Nginx es un servidor de tipo asíncrono, al contrario que Apache, que funciona en modo síncrono.
En modo síncrono, se asigna un proceso a cada cliente, lo que consume muchos recursos cuando hay muchas peticiones.
En modo asíncrono, un mismo proceso gestiona simultáneamente varios clientes, lo que limita el consumo de memoria y mejora el tiempo de respuesta.
No es preciso cargar en memoria los módulos necesarios para el proceso del cliente.
Nginx es un programa open source, con una licencia de tipo BSD.
Es compatible con Linux, BSD, OSX y Windows, pero generalmente se usa en plataformas Linux o BSD.
2. Archivo de configuración
El archivo de configuración de Nginx se creó para ser más sencillo y más fácil de mantener que el de Apache.
Sin embargo, su simplicidad implica un gran rigor, ya que un error de sintaxis puede bloquear el arranque del servidor (y también impedir su parada).
a. Formato del archivo de configuración
El archivo de configuración de Nginx por defecto es nginx.conf.
Está en el repertorio raíz definido en la instalación del programa, generalmente /etc/nginx.
Su estructura es diferente de la del archivo de configuración de Apache y está menos documentado.
Como Nginx gestiona menos opciones que Apache, el archivo de configuración será más pequeño y más fácil de mantener.
Algunas directivas están integradas en contenedores para que su ámbito de aplicación sea limitado.
*Formato tipo del archivo de configuración Nginx
directiva1 valor1 ; 
directiva2 valor2 ; 
# Comentario 
Bloque1 {   
  directiva3 valor3 ;  
  directiva4 valor4 ;  
  ...   
}  
...  
BloqueM {   
     directiva3 valor3 ; 
    BloqueN { 
  directivaN valorN ; 
    }  
  ...   
}
Podemos distinguir:
- Directivas incluidas directamente en el archivo de configuración, fuera de cualquier bloque. Una directiva termina obligatoriamente con un carácter ;.
- Directivas integradas en un bloque de directivas, delimitado por llaves.
- Líneas de comentarios, cuyo primer carácter es una almohadilla.
- Las directivas que están fuera de los bloques (a veces se puede decir que están en el bloque cero) son generales y se aplican al servidor en su conjunto, excepto a los bloques que dan otro valor para esta misma directiva.
- Las directivas definidas en un bloque se aplican al propio bloque y a los bloques que contiene, si estos no dan otro valor para esta directiva.
- Se dice que la directiva más cercana al bloque actual «oculta» las anteriores.
Muchas directivas son opcionales y el servidor utiliza un valor por defecto si no están en el archivo de configuración.
Estos valores por defecto son los más adecuados en la mayoría de los casos, lo que permite limitar el número de líneas del archivo de configuración.
b. Directivas generales
Las siguientes directivas definen algunos parámetros generales y deberían estar en cualquier configuración Nginx.
*Archivo de configuración: directivas elementales
user Nombre Grupo --> Designa la cuenta de servicio propietario y el grupo asociado a los procesos Nginx.
worker_processes Número --> Número de procesos de trabajo. El valor recomendado es de uno por procesador.
include RutaArchivo --> Indica un archivo de configuración anexo que se integra en el archivo de configuración.
*Los bloques de directivas
La mayor parte de las directivas se incluyen en los bloques de directivas.
Un bloque puede contener otros bloques.
Atención, Nginx distingue mayúsculas en las directivas: Include con una I mayúscula provocará un error en la carga.
*Sintaxis genérica de un bloque de directiva
NombreBloque [argumento bloque] { 
  directiva valor 
  directiva valor 
  ... 
}
Al contrario que en Apache, no existen directivas generales (fuera de bloques) que permitan especificar las direcciones de escucha (Listen) ni el directorio raíz de las páginas del servidor por defecto (DocumentRoot).
*Archivo de configuración mínimo
Por defecto, después de la instalación del paquete, Nginx utiliza parámetros por defecto, que se han puesto en la compilación del programa.
Por ejemplo, para una instalación CentOS reciente, tendremos como parámetros por defecto:
- Cuenta de usuario nginx
- Grupo nginx
- Escucha en el puerto 80 para todas las direcciones de la máquina.
- /etc/nginx/html/ como raíz para las páginas web.
- index.html como página de inicio por defecto.
Estos parámetros no corresponden necesariamente con la configuración del sistema donde se instala.
En el ejemplo CentOS, el directorio /etc/nginx/html/ no existe y la página de inicio instalada por defecto con el paquete está en /usr/share/nginx/html.
Por lo general, hay que colocar de forma adecuada los parámetros básicos para poder iniciar correctamente el servidor Nginx.
Este archivo de configuración mínimo que permite iniciar el servicio y mostrar la página de inicio por defecto en un navegador es el siguiente:
events { 
}  
  
http { 
    server { 
        root  /usr/share/nginx/html; 
    } 
}
El primer bloque tiene como etiqueta events y su contenido está delimitado por llaves. Este bloque es obligatorio.
El bloque http define las características del rol de Nginx como servidor HTTP (sabiendo que puede jugar otros roles y que a veces no se utiliza para HTTP).
El bloque server está dentro del bloque http y permite definir un servidor HTTP.
En el bloque server, encontramos una directiva root, que especifica la ruta de acceso al directorio raíz del servidor definido por este bloque, que contiene los directorios y las páginas.
Este servidor acepta cualquier petición de conexión en el puerto 80, para todas las direcciones IP de las interfaces del equipo, y muestra las páginas desde el directorio raíz /usr/share/nginx/html.
La página de inicio por defecto será /usr/share/nginx/html/index.html.
Este archivo de configuración generalmente es insuficiente y se debe completar para permitir especificar las características reales de funcionamiento del servicio.
*Ejemplo de archivo de configuración mínima mono servidor
user  nginx nginx; 
events { 
   worker_connections  1024; 
} 
http { 
    include /etc/nginx/mime.types; 
    index index.html index.htm; 
    server { 
        listen 80; 
        server_name localhost; 
        root  /usr/share/nginx/html; 
    } 
}
La primera directiva user, fuera de cualquier bloque, especifica el usuario y el grupo asociados a los procesos servidores de Nginx (excepto el proceso inicial, que siempre está asociado al super-user root).
La cuenta de usuario y el grupo deben existir.
El primer bloque tiene como etiqueta events.
La directiva worker_connections indica el número de conexiones que puede gestionar simultáneamente un proceso servidor.
Este bloque es obligatorio.
El bloque http define las características del rol de Nginx como servidor http (sabiendo que puede jugar otros roles y que a veces no se utiliza para HTTP).
La primera directiva, include, permite incluir, en lugar de esta línea, uno o varios archivos de configuración, en este caso el archivo por defecto de declaración de tipos de archivos MIME.
La directiva index permite indicar los nombres por defecto de la página de inicio de todos los servidores.
Los nombres se buscan en el orden indicado y el primer archivo que corresponde a un nombre de la lista se carga como página de inicio si no se ha especificado nada en la petición.
Una directiva se aplica al bloque que la contiene, así como, por defecto, a todos los bloques contenidos en ese bloque.
El bloque server siempre se incluye dentro del bloque http y permite definir un servidor HTTP.
La directiva listen indica en qué direcciones o números de puerto el servidor se pone a la escucha de peticiones de conexión.
En el ejemplo, este servidor se asociará a todas las peticiones de conexión que lleguen al puerto 80 de cualquier dirección IP del equipo.
La directiva server_name declara el nombre del servidor.
En el ejemplo, este servidor se asociará a cualquier petición de un navegador que se haga con el nombre localhost.
La directiva root especifica la ruta de acceso al directorio raíz que contiene las páginas asociadas a este servidor.
En este ejemplo, si el navegador pide la página de inicio de la máquina beta, funcionará.
Como no hay ningún bloque servidor que esté asociado a este nombre, Nginx utiliza el primer bloque servidor para la escucha en la dirección y el puerto correspondientes.
c. Reglas de sintaxis
La sintaxis del archivo de configuración de Nginx es bastante simple, pero debemos ser rigurosos.
Sobre todo:
- Distinción entre mayúsculas y minúsculas: la mayor parte de estas directivas se deben indicar en minúsculas.
- Separador de palabras para una directiva: espacio(s) o tabulación(es). Si una cadena de caracteres contiene caracteres especiales (espacio, llave o punto y coma), hay que ponerla entre comillas simples o dobles.
- Una directiva siempre termina con un punto y coma.
- Un comentario se escribe comenzando con una almohadilla.
- Un bloque se delimita con una llave de apertura y una llave de cierre. Las sangrías, mediante espacios o tabulaciones, son opcionales (pero se aconseja para una mejor legibilidad).
d. Validación de la sintaxis
Si hay un error de sintaxis en el archivo de configuración, Nginx no se iniciará.
Además, en algunos casos, si el archivo de configuración se modifica mientras el servidor Nginx está funcionando y hay un error de sintaxis, el servidor no se puede parar.
Así pues, es prudente validar la sintaxis del archivo de configuración antes de iniciar el servicio.
*Validación de la sintaxis del archivo nginx.conf
nginx -t
*Ejemplo
root@alpha:~# cat /etc/nginx/nginx.conf 
user  nobody nogroup; 
events { 
   worker_connections  1024; 
} 
http { 
        ######## ERROR DE SINTAXIS VOLUNTARIO ############# 
    Include /etc/nginx/mime.types; 
        ######## ERROR DE SINTAXIS VOLUNTARIO ############# 
    index index.html index.htm; 
    server { 
        listen 80; 
        server_name localhost; 
        root  /usr/share/nginx/html; 
    } 
} 
root@alpha:~# nginx -t 
nginx: [emerg] unknown directive "Include" in /etc/nginx/nginx.conf:7 
nginx: configuration file /etc/nginx/nginx.conf test failed
*Después de la corrección del error:
root@alpha:~# nginx -t 
nginx: the configuration file /etc/nginx/nginx.conf syntax is ok 
nginx: configuration file /etc/nginx/nginx.conf test is successful
e. Configuración por defecto de tipo Debian
Las distribuciones Debian o Ubuntu configuran Nginx por defecto utilizando diferentes subdirectorios para los servidores virtuales.
En el directorio /etc/nginx, se crean dos directorios: sites-available y sites-enabled.
El archivo de configuración principal es /etc/nginx/nginx.conf y contiene una directiva que incluye automáticamente todos los archivos situados en sites-enabled:
include /etc/nginx/sites-enabled/*;
Para organizar los diferentes servidores virtuales, basta con crear un archivo de configuración por servidor virtual en el directorio sites-available.
Para activar un servidor virtual, habrá que crear en el directorio sites-enabled un enlace a ese archivo de configuración.
Para desactivarlo, habrá que eliminar el enlace.
La configuración inicial utiliza este mecanismo para el servidor por defecto:
root@alpha: # ls -l /etc/nginx/sites-available/ 
-rw-r--r-- 1 root 2765 nov.  20  2013 default 
root@alpha: # ls -l /etc/nginx/sites-enabled 
lrwxrwxrwx 1 root root 34 jul.  1 18:55 default -> 
/etc/nginx/sites-available/default

3. Los módulos Nginx
a. Carga de los módulos
El servidor web Nginx tiene una estructura modular, es decir, que las funciones principales del servidor están en un módulo principal (core), mientras que otras funciones están en módulos distintos.
Esta arquitectura permite facilitar el desarrollo y el mantenimiento de la aplicación, y optimizar el consumo de recursos, seleccionando solo los módulos necesarios.
Al contrario que Apache, que dispone de módulos dinámicos, los cuales se pueden cargar a petición indicándolo en el archivo de configuración, los módulos de Nginx son estáticos.
Se deben especificar en la compilación del ejecutable.
Esta elección está motivada por el rendimiento y la simplicidad del código.
Si instala Nginx utilizando paquetes binarios, debe comprobar que contienen todos los módulos que necesite.
Si no es así, estará obligado a descargar una versión del código fuente de Nginx y compilarla con los módulos que necesite.

c. Elección de los módulos
Los módulos dependen, por supuesto, del uso que se haga del servidor.
Esto consiste en identificar las necesidades y en determinar a través de la documentación en línea de Nginx cuáles son los módulos implicados.
Los módulos básicos son necesarios para el funcionamiento mínimo de Nginx y siempre están incluidos en el ejecutable:
1) Core module: núcleo de Nginx, gestiona la arquitectura general.
2) Event module: funciones de gestión básica de la red.
3) Configuration module: gestión de las inclusiones de archivos para la configuración.
Los módulos estándar implementan grandes categorías de funcionalidades de Nginx y generalmente se incluyen en el ejecutable, excepto en algún caso particular.
*A continuación podemos ver los principales:
 - HTTP core: gestiona el servidor HTTP de Nginx
 - Access: control de acceso según la dirección IP
 - Auth Basic: autentificación básica
 - Auto Index: gestión de listas de directorios predeterminados
 - Browser: gestión del tipo de navegador cliente
 - Charset: codificación de caracteres de las páginas
 - FastCGI: gestión de scripts FastCGI (PHP, Python, Perl, etc.)
 - Gzip: compresión
 - Index: gestión de la página de inicio predeterminada
 - Limit: control de recursos
 - Log: gestión de logs
 - Rewrite: permite especificar directivas de sobre escritura de URI
 - SSI: gestión de SSI
*Otros módulos son opcionales, según las necesidades identificadas, por ejemplo:
 - Proxy: gestión del rol de Proxy
 - Upstream: equilibrio de carga
 - SSL: gestión del protocolo SSL
 - Mail Core: servidor de correo mail
 - POP3: gestión del servidor POP3
 - IMAP: gestión del servidor IMAP
 - SMTP: gestión del servidor SMTP
Finalmente, existen numerosos módulos de terceros, es decir, no soportados oficialmente por los creador de Nginx, por ejemplo:
 - Access Key: control de acceso por una clave en la URI.
 - Auth PAM: autentificación por el módulo de sistema PAM.
 - GeoIP: geolocalización a partir de la dirección IP.
 - Mongo: implementación de la base de datos MongoDB.
 - UnZip: descompresión de archivos sobre la marcha.
4. Gestión de los recursos
El servidor Nginx generalmente tiene un mejor rendimiento que Apache y consume menos recursos de memoria.
Su arquitectura se concibió para ello, en detrimento de la elección de funcionalidades (menos módulos disponibles, necesidad de compilarlos con el ejecutable).
Escrito en lenguaje C, multiproceso pero monohilo, permite obtener un rendimiento excelente.
Gracias a su gestión de las entradas/salidas y al hecho de que un proceso pueda gestionar simultáneamente n conexiones.
*Observemos los procesos ejecutados por Nginx en un servidor inactivo:
[root@beta nginx]# ps -ef | grep nginx  
root     10220     1  0 17:31 ?        00:00:00 nginx: master 
process /usr/sbin/nginx -c /etc/nginx/nginx.conf  
nginx    10222 10220  0 17:31 ?        00:00:00 nginx: worker process
Podemos constatar la presencia de un proceso raíz asociado a root y otro proceso asociado a la cuenta de servicio nginx.
El primer proceso solo sirve para ejecutar el resto, que tratará las peticiones de los clientes.
El servidor que podemos ver aquí no gestiona ninguna conexión con ningún cliente; es un proceso de trabajo preasignado, que está preparado para gestionar cualquier petición de clientes.
La gestión del primer servicio de la cuenta root es obligatorio, ya que es el único autorizado a abrir el puerto 80 en un sistema Linux.
Se recomienda tener tantos procesos de trabajo como procesadores.
El número se configura mediante la directiva general worker_processes.
Se puede especificar la palabra clave auto en lugar de un número, en cuyo caso Nginx creará tantos procesos de trabajo como procesadores detectados.
*Ejemplo
[root@beta ~]# cat /etc/nginx/nginx.conf 
user  nginx nginx; 
worker_processes  2; 
events { 
    worker_connections  1024; 
} 
 
http { 
    include /etc/nginx/mime.types; 
    index index.html index.htm; 
    server { 
        listen 80; 
        server_name localhost; 
        location / { 
            root  /usr/share/nginx/html; 
        } 
    } 
}
*Observemos los procesos ejecutados por Nginx en este servidor inactivo:
root     10271     1  0 17:37 ?        00:00:00 nginx: master 
process /usr/sbin/nginx -c /etc/nginx/nginx.conf 
nginx    10272 10271  0 17:37 ?        00:00:00 nginx: worker process 
nginx    10274 10271  0 17:37 ?        00:00:00 nginx: worker process
Hay dos procesos de trabajo.
Al contrario que Apache, Nginx no inicia tantos procesos como conexiones.
El número de procesos de trabajo permanece estable, ya que cada uno puede gestionar un gran número de conexiones.
Este mecanismo contribuye al buen rendimiento de Nginx, así como a un consumo limitado de memoria.
El número máximo de conexiones simultáneas por proceso de trabajo se define mediante la directiva worker_connections del bloque events:
events { 
    worker_connections  1024; 
}
1024 conexiones simultáneas por proceso de trabajo.
5. Nginx y las expresiones regulares
Nginx acepta expresiones regulares como parámetro de numerosas directivas de configuración.
Las expresiones regulares que se pueden utilizar son las del lenguaje Perl, implementadas por la biblioteca PCRE (Perl Compatible Regular Expression).
Esta implementación es particularmente rica y ofrece un lenguaje completo de descripción de patrones de cadenas de caracteres (patterns).
Las expresiones regulares constituyen un verdadero lenguaje, complejo y potente, analizado por un programa especializado (motor de análisis).
Una expresión regular es un patrón (pattern) que describe un conjunto de cadenas de caracteres con las mismas características.
Para definir el patrón, se utiliza una combinación de caracteres literales y de caracteres que tienen un significado especial, los metacaracteres, que se pueden completar con cuantificadores y modificadores.
Comparando una cadena de caracteres con un patrón, solo se pueden obtener dos resultados: la cadena corresponde al patrón o no corresponde.
Así se pueden seleccionar cadenas según los criterios representados por este patrón.
*Ejemplo
La expresión regular www corresponde a todas las cadenas de caracteres que contienen los caracteres www, como las siguientes cadenas:
www 
wwww 
www.ediciones-eni.com 
wwworld.com
Pero no las siguientes cadenas:
WWW 
wwt 
wwew 
ftp
Los metacaracteres son específicos de las expresiones regulares y no tienen el mismo sentido que los que se encuentran en los especificadores de nombres de archivos (caracteres comodín, o globs en términos Unix).
Atención, pues, a no confundirlos.
Hay que insistir particularmente en el carácter * que, en una expresión regular, no significa una relación de caracteres cualesquiera, como es el caso en un nombre de archivo en el shell.
Las expresiones regulares están formadas por patrones que permiten describir un conjunto de caracteres.
Estos patrones se componen de caracteres literales, es decir, sin un significado en particular, de metacaracteres, de cuantificadores y de modificadores.
*Caracteres literales
Son caracteres que no tienen un significado específico para el motor de análisis de las expresiones regulares. 
*Ejemplo:
En la siguiente expresión regular:
conta
Las siguientes cadenas corresponden:
www.conta.masoc.com
www.conta.masoc.com/info
www.masoc.es/info/contabilidad.html
Las siguientes cadenas no corresponden:
www.cuenta.masoc.es
Cuenta.masoc.com
*Metacaracteres
Los metacaracteres tienen un significado particular para el motor de expresiones regulares.
Como son muy numerosas, nos limitaremos a mencionar las más habituales:
. --> Un solo carácter cualquiera.
 --> Inicio de una cadena.
$ --> Final de una cadena.
[abc] --> Un solo carácter perteneciente a la lista entre corchetes.
\car --> Quitar el significado especial al metacaracter car.
Atención al carácter . en las URI. 
Como se considera un metacarácter, hay que poner el prefijo \ cuando se quiere incluir el carácter . en una expresión regular:
*Ejemplos:
En la siguiente expresión regular:
www.masoc.com
Las siguientes cadenas corresponden:
www.masoc.com
www.masoc.com/
www/masoc/com
wwwrmasocvcom
www masoc com
Las siguientes cadenas no corresponden:
www.tasoc.com
WWW.masoc.com
/masoc
En la siguiente expresión regular:
www\.masoc\.com
Las siguientes cadenas corresponden:
www.masoc.com
www.masoc.com/
Las siguientes cadenas no corresponden:
www/masoc/com
wwwrmasocvcom
www masoc com
www.tasoc.com
WWW.masoc.com
/masoc
En la siguiente expresión regular:
[mM]asoc\.com$
Las siguientes cadenas corresponden:
www.masoc.com
www.Masoc.com
www.mamasoc.com
ftp.masoc.com
Las siguientes cadenas no corresponden:
www.masoc.com/
www.MASOC.com/
En la siguiente expresión regular:
www\.masoc\.com$
La siguiente cadena corresponde:
www.masoc.com
Las siguientes cadenas no corresponden:
www.masoc.com/
masoc.com
www1.Masoc.com
www.mamasoc.com
ftp.masoc.com
*Cuantificadores elementales
Un cuantificador es un carácter especial que permite especificar el número de repeticiones del carácter (a veces de un grupo de caracteres) que le precede.
Por sí mismo no tiene significado y siempre hay que interpretarlo en combinación con otros caracteres.
A continuación veremos los cuantificadores más sencillos y los más habituales:
* --> Repetición de 0 a n veces del carácter precedente.
+ --> Repetición de 1 a n veces del carácter precedente.
? --> Repetición de 0 o 1 veces del carácter precedente.
\{n\} --> Repetición de n veces del carácter precedente.
\{n,m\} --> Repetición de n a m veces del carácter precedente.
*Ejemplo:
En la siguiente expresión regular:
go+gle
Las siguientes cadenas corresponden:
google 
gogle 
www.google.com 
goooooooooooogle
Las siguientes cadenas no corresponden:
goagle 
ggle
En la siguiente expresión regular:
go?gle
Las siguientes cadenas corresponden:
gogle 
google 
www.google.com 
google is your friend
Las siguientes cadenas no corresponden:
gooogle 
goooooooooooogle
En la siguiente expresión regular:
[rb]{1,3}oot$
Las siguientes cadenas corresponden:
root 
boot 
bboot 
rrroot 
rbroot
Las siguientes cadenas no corresponden:
rte 
boooot 
Root
El cuantificador *
Hay que insistir en este cuantificador, ya que a menudo es origen de muchos errores.
Indica que el carácter (o grupo de caracteres) que le precede se puede repetir de 0 a n veces en las cadenas que corresponden al patrón.
Por lo tanto, la ausencia del carácter precedente corresponde al caso de la repetición de 0 veces.
*Ejemplo:
En la siguiente expresión regular:
go*gle
Las siguientes cadenas corresponden:
google 
ggle 
goooooogle 
www.goooooooooooogle.com
Las siguientes cadenas no corresponden:
gOogle 
goagle 
goooooOgle 
gooooozgle
Para especificar un conjunto de caracteres cualesquiera (excepto el fin de línea), incluida la opción de no poner ningún carácter, lo que corresponde al sentido habitual del asterisco,
habrá que utilizar la siguiente combinación:
.* --> Un carácter cualquiera (.), repetido de 0 a n veces (*)
*Ejemplo:
En la siguiente expresión regular:
r.*t
Las siguientes cadenas corresponden:
www.brots.com 
www.masoc.com/root 
chroot 
www.deroooooooooooote.org 
www.masoc.com/carta.php 
rt
Las siguientes cadenas no corresponden:
/rojo 
boot.php 
12345.html
