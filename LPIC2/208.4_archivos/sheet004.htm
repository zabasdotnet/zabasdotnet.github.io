Los filtros de URI de Nginx: el bloque de tipo ubicación Volver
Un servidor Nginx utiliza bloques de tipo ubicación para asociar características de procesos a las URI que pide el cliente.
Un bloque de tipo ubicación se puede aplicar a toda o parte de la URI, puede corresponder a una sola URI o a un conjunto de URI.
Nginx utiliza una sintaxis flexible y potente para definir los patrones de URI, incluida la posibilidad de especificarlas con expresiones regulares.
Este mecanismo permite gestionar tanto directorios como tipos de archivos.
*Por ejemplo:
- se puede asociar restricciones de acceso a ciertas carpetas de un sitio virtual
- se puede redirigir las peticiones de los archivos PHP a un servidor remoto
- se puede comprimir sobre la marcha archivos de tipo imagen de un servidor virtual concreto.
Los filtros URI son muy potentes, pero pueden resultados complicados de configurar.
De hecho, una URI puede corresponder a diferentes filtros, aunque solo se aplicará uno, según las reglas de prioridad.
Más tarde, emplearemos a menudo el término inglés location para designar un filtro URI.
URI (Uniform Resource Identifier) es la forma de designar un recurso de Internet, por ejemplo:

1. Definición de un bloque de ubicación de selección de URI
Los bloques de tipo ubicación deben estar obligatoriamente dentro de un bloque server.
Pueden ser varios y de diferentes tipos.
Cuando el servidor Nginx recibe una petición para un servidor, este decodifica la URI (sustituyendo los caracteres especiales por su valor, por ejemplo %20 se sustituirá por un espacio).
A continuación se aísla la parte central de la URI, entre el nombre del servidor y el final de la URI, o antes de los caracteres ? o # que preceden a los parámetros.
Se compara esta cadena de caracteres con las diferentes directivas de ubicación, para seleccionar la que corresponde a la URI.
a. Sintaxis
La sintaxis de definición del bloque es la siguiente:
location Expresión_Selection { 
Directivas 
}
*La expresión de selección de las URI puede utilizar diferentes formas:
- location = Cadena
Una URI se asociará a este bloque si corresponde exactamente a la cadena de caracteres especificada.
- location Cadena
Una URI se asociará a este bloque si comienza por la cadena de caracteres especificada.
- location ~ Expresión regular
Una URI se asociará a este bloque si corresponde (matching) a la expresión regular especificada (con distinción de mayúsculas).
- location ~* Expresión regular
Una URI se asociará a este bloque si corresponde (matching) a la expresión regular especificada (sin distinción de mayúsculas).
- location ˆ~ Cadena
Una URI se asociará a este bloque si comienza por la cadena de caracteres especificada.
Si esta cadena de caracteres corresponde, se utilizará, pero el operador ˆ~ desactiva el tratamiento de bloques con expresiones regulares.
b. Prioridad de selección
Los diferentes bloques de ubicación del servidor virtual se aplican a la URI en el siguiente orden:
1) Ubicación con el operador =
Si una URI corresponde exactamente, la búsqueda se para.
Se utilizará esta ubicación.
2) Ubicación con cadenas de caracteres
Si una URI corresponde y el patrón tiene un prefijo ˆ~, la búsqueda se parará.
Después evaluará ubicaciones con expresiones regulares.
3) Ubicación con expresión regular
Son las últimas en evaluarse.
La primera expresión regular que corresponda a la URI para la búsqueda.
Una vez terminada la búsqueda, se aplican las siguientes reglas de precedencia:
Si una ubicación con el operador = corresponde exactamente a la URI, es esta la que se utiliza.
Si no
Si una ubicación con una cadena de caracteres simple corresponde exactamente a la URI, es esta la que se utiliza.
Si no
Si una ubicación con el operador ˆ~ corresponde al inicio de la URI, es esta la que se utiliza.
Si no
Si las ubicaciones con expresión regular corresponden a la URI, la que se utiliza es la primera que se encuentra en el orden del archivo de configuración.
Si no
Si las ubicaciones con cadena de caracteres simple corresponden al inicio de la URI, la que se utiliza es la que corresponda a la parte más grande de la URI.
c. Ejemplos de selección
El orden de los bloques de ubicación dentro de un bloque servidor no es necesariamente significativo, excepto en el caso de ubicaciones múltiples con expresiones regulares, ya que se elegirá la primera que corresponda a la URI.
Como las reglas de selección del bloque de ubicación son complejas, es conveniente hacer pruebas completas para validar la configuración.
*Ejemplo
Bloque server en el archivo de configuración:
server {  
        listen       80 ;  
        server_name localhost;  
        root   /usr/share/nginx/html;  
        location = / {  
            index index_root.html;  
        }  
        location /string {  
            index index_string.html;  
        }  
        location ˆ~ /stringnoreg {  
            index index_noreg.html;  
        }  
        location /stringnoreg_longer {  
            index index_string.html;  
        }  
        location ~ reg/$ {  
            index index_reg.html;  
        }  
        location ~ string.*reg/$ {  
            index index_reg1.html; 
        } 
}
Dos ubicaciones son de tipo expresiones regulares:
 - reg/$  --> cualquier URI que termine con reg/ (el carácter $ representa el fin de la cadena de caracteres).
 - string.*reg/$   --> cualquier URI que contenga el conjunto de caracteres string y que termine con reg/ (.* significa un conjunto cualquiera de caracteres, incluida la opción de ningún carácter y el carácter $ representa el fin de la cadena de caracteres).
*Arborescencia del servidor:
Hay siete directorios en el directorio raíz del servidor, /usr/share/nginx/html:

stringnoregreg

En cada directorio, hay cinco archivos de índice:

index_string.html

Como la sintaxis de los bloques de ubicación es delicada, a menudo es muy útil consultar el archivo log de errores durante las pruebas.
La ubicación de este archivo se define en el archivo de configuración y por defecto es /var/log/nginx/error.log.
2. Bloque de ubicación con nombre
Este tipo de ubicación lo utiliza exclusivamente en modo interno el servidor nginx para gestionar redirecciones de URI generadas por otras directivas (try_files, error_page).
El operador es el carácter @.
*Ejemplo
location @NotFound { 
# directivas 
}
Definición de un bloque de ubicación llamado NombreBloque, que se podrá llamar por medio de algunas directivas (try_files, error_page):
error_page 404 = @NotFound;
En este ejemplo, si hay una petición a una página que no existe (error 404), la URI se transmitirá al bloque llamado NotFound.
