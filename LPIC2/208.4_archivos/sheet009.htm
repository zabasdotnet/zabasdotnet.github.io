Nginx como reverse proxy Volver
Es difícil migrar un servidor web Apache que ya exista a Nginx.
La configuración de Apache suele ser compleja y no es fácil reproducirla en Nginx, ya que los servidores tienen una concepción muy diferente.
En particular, Nginx no permite establecer archivos de configuración a nivel de directorio, los archivos .htaccess de Apache.
Una solución que se utiliza frecuentemente es configurar Nginx como reverse proxy del servidor Apache.
Esto permite (casi) no modificar nada en la configuración del servidor Apache al tiempo que nos beneficiamos del excelente rendimiento de Nginx como servidor web estático.
1. Reverse proxy
Un reverse proxy HTTP (que se puede traducir como proxy inverso) es un servidor HTTP que recibe las peticiones de los clientes y las envía a uno o varios servidores HTTP.
Esta técnica se utiliza con diferentes objetivos:
- Hacer que servidores HTTP situados en la red interna de la empresa sean accesibles desde el exterior:
Solo el reverse proxy HTTP (en la DMZ) es visible desde el exterior y redirige las peticiones a los servidores internos.
- Servidor de caché:
Como el reverse proxy recibe todas las peticiones de los clientes, de uno o varios servidores, puede poner en la caché los elementos no dinámicos que se pidan.
Mejora el rendimiento y reduce la carga de los servidores finales.
- Reparto de carga (load balancing):
El reverse proxy puede gestionar un pool de servidores HTTP de aplicaciones y repartir las consultas de los clientes, según diferentes criterios, hacia los diferentes elementos del pool.
Nginx se puede utilizar para todas estas funciones.
Como complemento del rol de servidor de caché, Nginx también se puede configurar para gestionar todas las peticiones no dinámicas (páginas estáticas, archivos de imágenes, vídeos, hojas de estilo, bibliotecas javascript, etc.).
También para redirigir las peticiones dinámicas (scripts PHP, Python…) a uno o varios servidores.
Como su gestión de acceso a discos y a la caché de memoria es especialmente eficaz, Nginx permitirá mejorar de forma muy significativa el rendimiento de los servidores Apache que tengan muchas peticiones.
El servidor Apache tendrá muchas menos peticiones y solo se le enviarán las de los elementos dinámicos.
Nginx también dispone de mecanismos flexibles y potentes de balanceo de carga.
En el ámbito del examen LPIC2, es importante conocer la configuración básica de Nginx como reverse proxy.
2. El módulo ngx_http_proxy
Este módulo estándar está incluido por defecto en el servidor Nginx (excepto si se compila con la opción --without-http_proxy_module).
Proporciona un conjunto variado de directivas para gestionar y optimizar las redirecciones de consultas, así como la puesta en caché de los elementos devueltos.
Aquí nos limitaremos a las directivas básicas para implementar Nginx como reverse proxy.
3. Declaración del servidor de destino
La directiva proxy_pass permite indicar a Nginx hacia que servidor debe redirigir una petición.
La sintaxis completa es:
proxy_pass Protocolo://Host[:Puerto][/URI]
*Con:
Protocolo --> puede tratarse de http o https para redirigir en SSL.
Host --> se puede especificar el nombre del host o su dirección IP o ruta de acceso de socket Unix con el prefijo unix:.
Puerto --> solo es obligatorio si el servidor de destino está en la misma máquina (se deberá configurar con un número de puerto diferente del asignado a Nginx).
URI --> opcional. La parte de la URI de origen correspondiente a la ubicación especificada (si es preciso modificada por una directiva) se remplazará por la cadena URI indicada y a continuación se enviará al servidor destino.
*Ejemplos:
proxy_pass http://10.0.0.2;
Redirección al servidor remoto.
proxy_pass http://www.masoc.local;
Redirección al servidor remoto.
proxy_pass https://10.0.0.2;
Redirección al servidor remoto en SSL.
proxy_pass http://127.0.0.1:8080;
Redirección al servidor local de la máquina, que escucha en el puerto 8080.
location /info/ { 
proxy_pass http://www.masoc.local/externo/; 
}
Redirección al servidor remoto, sustituyendo /info/ en la URI por /externo/.
4. Selección de las peticiones que se deben redirigir
Otras técnicas permiten determinar las peticiones que se han de tratar localmente por el servidor Nginx y las peticiones que se han de redirigir a un servidor de destino.
a. Selección mediante un bloque de ubicación utilizando una expresión regular
Se puede definir un bloque de ubicación específico para cada tipo de petición que se ha de redirigir.
*Ejemplo
Para redirigir solo las peticiones de los scripts PHP (archivo con sufijo .php, .php3, .php4, .php5, etc.) a un servidor remoto:
http { 
    include /etc/nginx/mime.types; 
        index index.html index.htm; 
        server { 
          listen 80 default; 
          server_name beta; 
          root  /usr/share/nginx/html/public; 
          location ~* \.php.*$ { 
             proxy_pass http://www.local; 
          } 
        } 
}
b. Selección mediante la directiva try_files
Esta directiva permite especificar diferentes criterios de búsqueda del objeto que se pide, utilizado en el orden indicado.
Al final de la directiva, se puede especificar un nombre de bloque de ubicación con nombre.
Así, primero se puede intentar servir localmente las peticiones y a continuación, en caso de fallo, redirigir la petición a un servidor destino.
*Ejemplo:
http {  
    include /etc/nginx/mime.types;  
    index index.html index.htm;  
    server {  
        listen 80 default;  
        server_name beta;  
        root  /usr/share/nginx/html/public;  
        location @destino {  
            proxy_pass http://www.local;  
        }  
        location / {  
            try_files $uri $uri/ @destino;  
        }  
    }  
}
Cualquier petición se buscará a partir de la raíz del servidor.
Si no se encuentra el elemento correspondiente, se pone el sufijo / a la URI.
Si tampoco se encuentra, se redirige la petición al bloque de ubicación llamado destino, definido más arriba y asociado al servidor destino www.local.
La variable $uri contiene la URI que pide el cliente.
c. Selección mediante la directiva fastcgi_pass
La directiva fastcgi_pass, del módulo estándar http_fastcgi, permite asociar un bloque de ubicación a una redirección hacia un servidor de scripts remoto.
De nuevo, esto permite tratar los elementos estáticos en local por Nginx y redirigir las peticiones dinámicas a un servidor de destino.
La sintaxis completa es:
fastcgi_pass Host[:Puerto]
*Con:
Host  --> se puede especificar un nombre de host o su dirección IP o una ruta de acceso de socket Unix con el prefijo unix:.
Puerto  --> opcional.
http { 
    include /etc/nginx/mime.types; 
    index index.html index.htm; 
    server { 
        listen 80 default; 
        server_name beta; 
        root  /usr/share/nginx/html/public; 
        location @dyn { 
            fastcgi_pass www.local; 
        } 
        location / { 
            try_files $uri $uri/ @dyn; 
        } 
    } 
}
