Restricciones de acceso a usuarios en un servidor Nginx Volver
Como en Apache, Nginx ofrece numerosas posibilidades de restricciones de acceso a usuarios y existen numerosos métodos para gestionar el acceso a los datos.
Hay que considerar que este párrafo solo trata los métodos de autentificación de los usuarios, pero no los métodos de cifrado.
Es decir, que se puede autentificar el acceso a los datos, pero las páginas web se transmiten sin cifrar entre el servidor y el navegador.
Si queremos cifrar los datos, deberemos utilizar el protocolo SSL.
1. Control por dirección IP
El módulo de control de acceso, ngx_http_access, forma parte de los módulos del núcleo de Nginx y, por tanto, está incluido por defecto en el servidor.
(Excepto si explícitamente se excluyó de la compilación mediante la opción --without-http_ access_module).
Este módulo permite o prohíbe el acceso a elementos gestionados por el servidor Nginx, en función de la dirección IP de donde provenga la petición.
Se utilizan dos directivas, allow y deny, que respectivamente permiten o prohíben el acceso a los elementos del bloque en el que están definidos.
Puede haber varias en un mismo bloque.
En este caso se leen en el orden del archivo de configuración, y es la primera la que se aplica en el cliente.
Se pueden utilizar diferentes sintaxis para especificar los clientes afectados:
Dirección --> en formato IP v4 o v6. 
Subred --> en formato CIDR (Identificador/Número de bits del identificador de red). 
all  --> para todas las direcciones. 
unix: -->  para los sockets Unix.
Como en todas las directivas, las que están en un bloque ocultan las de bloques de nivel más alto.
*Ejemplo:
http {  
    include       /etc/nginx/mime.types;  
        index index.html index.htm;  
        allow 127.0.0.1;  
        allow 192.168.56.1;  
        deny 10.0.2.3;  
        allow 10.0.2.0/24;  
        deny all;  
        server {  
          listen 80 default;  
          server_name beta;  
          root  /usr/share/nginx/html/public;  
          }  
  
        server {  
          listen 80;  
          server_name www.conta.masoc.com;  
          root  /usr/share/nginx/html/public/conta;  
          allow 10.0.1.0/24;  
          deny all;  
          }  
        server {  
          listen 80;  
          server_name www.marketing.masoc.com;  
          root  /usr/share/nginx/html/public/mkg;  
          allow all;  
          }  
}
Por defecto, solo se podrá acceder a los servidores virtuales desde clientes de la máquina local, de la subred 10.0.2.0 (excepto 10.0.2.3) y del equipo 192.168.56.1.
Todo el mundo puede acceder al servidor virtual www.marketing.masoc.com.
Solo se puede acceder al servidor virtual www.conta.masoc.com desde la subred 10.0.1.0.
Una petición al servidor virtual beta/, que venga de un cliente cuya dirección sea 192.168.56.2, se rechaza con el siguiente mensaje:
Una directiva allow sola en un bloque no tiene sentido.
De hecho, autoriza la dirección indicada,  pero, sin directiva deny, el resto de las direcciones no están prohibidas.
Y si había directivas en un bloque de nivel superior, se anulan por el bloque que contiene la directiva allow:
http { 
    include       /etc/nginx/mime.types; 
        index index.html index.htm; 
        deny all; 
        server { 
          listen 80; 
          server_name www.marketing.masoc.com; 
          root  /usr/share/nginx/html/public/mkg; 
          # CONFIGURACIÓN ERRÓNEA VOLUNTARIA ################ 
          # Atención: abre el servidor a TODOS los clientes!!! 
          allow 50.0.0.1; 
          } 
}
En este ejemplo, mal configurado, todos lo clientes pueden acceder al servidor www.marketing.masoc.com, ya que la directiva anula la directiva deny all del nivel superior.
2. Control por autentificación

(Excepto si explícitamente se excluyó de la compilación mediante la opción --without-http_auth_basic_module).

Si en un bloque unimos el control de acceso por dirección y por autentificación, mediante la directiva satisfy podemos especificar si se deben cumplir las condiciones de los dos tipos o si basta una para validar el cliente:
- satisfy all;
La dirección Y la identidad deben cumplir las condiciones.
- satisfy any;
La dirección O la identidad deben cumplir las condiciones.
3. Control por autentificación local
Para implementar las restricciones de acceso, hay que definir qué elementos protegemos y con qué método.
Para los diferentes niveles sometidos a autentificación, hay que añadir las directivas necesarias según el modo de autentificación elegido.
Las siguientes directivas se aplican a la autentificación simple (local), gestionada por el módulo ngx_http_auth_basic.
a. Elección del ámbito de restricción de acceso simple
La restricción se puede hacer para todos los servidores, para un servidor virtual en concreto o para una URI que forme parte de los elementos gestionados por un servidor virtual.
Si queremos restringir el acceso a todos los sitios, basta con definir la restricción a nivel del bloque http.
De todos modos, al menos una página de inicio tiene que estar disponible para todo el mundo, por ejemplo la del sitio por defecto.
Si queremos restringir el acceso a un servidor, basta con definir la restricción de acceso a nivel del bloque server.
También se puede restringir el acceso a algunos elementos de un sitio, caso en el que definiremos la restricción de acceso a nivel de uno o varios bloques de tipo ubicación (ver más adelante).
b. Directivas de autentificación
En el bloque sometido a autentificación, hay que añadir las directivas necesarias para activar la autentificación e indicar la base de datos de cuentas local que hay que utilizar.
*Petición de autentificación local
auth_basic Mensaje_servidor;
Esta directiva provocará, en una petición de acceso al elemento, la visualización de un cuadro de diálogo de petición de autentificación.
El texto Mensaje_servidor se mostrará en el cuadro de diálogo.
Un campo nombre de usuario y campo contraseña permitirán al usuario introducir la información que se debe validar.
*Archivo de autentificación local
auth_basic_user_file Ruta_archivo;
Esta directiva especifica la ubicación del archivo que se debe utilizar para validar la petición de autentificación.
La ruta puede ser relativa al directorio del servidor Nginx (por defecto /etc/nginx) o absoluta.
El archivo indicado es un archivo de texto, con el mismo formato que el utilizado por Apache:
NombreUsuario : ContraseñaCifrada
El módulo de autentificación simple acepta varios tipos de cifrado de las contraseñas:
1) Función crypt(): se puede generar este tipo de contraseña utilizando el comando estándar de Apache:
                                       htpasswd o el proporcionado con openSSL: openssl passwd.
2) hash de tipo MD5: utilizado por algunas herramientas de Apache.
3) Modos RFC2307{TipoContraseña}Contraseña.
  El módulo gestiona los tipos PLAIN, SHA (1.3.13) y SSHA.
Podemos gestionar este archivo con el comando estándar Apache, htpasswd.
c. Creación de una base de datos de cuentas local
El comando htpasswd, proporcionado con el paquete Apache, permite crear fácilmente una base de datos de cuentas y alimentarla.
*Creación de la primera cuenta de usuario
htpasswd -c archivo nombre_usuario
*Añadir cuenta de usuario
htpasswd archivo nombre_usuario
*Eliminacion de una cuenta de usuario
htpasswd -D archivo nombre_usuario
*Ejemplo de archivo de contraseña
El archivo muestra en cada línea el nombre de la cuenta de usuario y su contraseña cifrada.
cat /usr/share/nginx/mdp/htpasswd 
garcia:WOVv7775MLoVc 
sancho:L03otawJqxXkE
*Ejemplo de servidor que pide autentificación
http { 
        include /etc/nginx/mime.types; 
        index index.html index.htm; 
          
        server { 
          listen 80; 
          server_name www.conta.masoc.com; 
          root  /usr/share/nginx/html/public/conta; 
          auth_basic "Sitio confidencial"; 
          auth_basic_user_file /usr/share/nginx/mdp/htpasswd; 
        }  
      }

4. Autentificación por LDAP
Nginx no dispone de ningún módulo estándar de gestión de la autentificación por LDAP.
No obstante, existen varias soluciones para gestionar una autentificación por LDAP a partir de diferentes módulos, opcionales o de terceros.
Se habrán de instalar e integrar con Nginx volviendo a compilar.
a. Utilización de PAM
Es posible pasar por la capa PAM (Pluggable Authentication Modules) del sistema para gestionar la autentificación por LDAP.
En este caso, hay que instalar el módulo ngx_http_auth_pam.
Hay que configurar PAM con un módulo específico para Nginx, que utiliza LDAP, y a continuación configurar la autentificación en el archivo de configuración Nginx.
b. Subconsulta
También se puede utilizar el módulo opcional ngx_http_auth_request, que permite delegar la autentificación a través de una subconsulta.
De este modo, se puede validar la petición mediante un script PHP u otro, local o remoto, accediendo a un servidor LDAP.
c. Módulo LDAP
Existe un módulo de un tercero, nginx-auth-ldap, instalable a partir del código fuente, que hay que compilar para generar el módulo.
